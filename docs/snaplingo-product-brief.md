## Problem Statement
People who work across languages all day do not want to context-switch into a full translator app. They want to select text anywhere on the desktop, trigger one fast action, and immediately get either:

1. a faithful translation they can trust, or
2. a more natural rewrite they can paste back with one click.

The product is not "another translator." The product is "the fastest way to turn selected text into usable text without breaking flow."

## What Makes This Cool
- It collapses three separate actions into one surface: capture selected text, translate it, and polish it for the target context.
- It feels native and invisible. The best session is a two-second detour, not an app launch.
- "Optimize" creates a real wedge. Users often need "make this sound natural in English" more than literal translation.

## Constraints
- Start with macOS only. Desktop selection is OS-specific, and cross-platform too early will slow the core loop.
- Optimize for short text snippets first: messages, emails, docs, subtitles, UI copy, and web paragraphs.
- The app must feel instant. A useful target is first result on screen in under 700 ms for translation and under 1500 ms for optimization on a normal network.
- v1 should work from a global hotkey and a small floating panel. Do not depend on users changing their writing workflow.
- v1 should avoid document translation, OCR, browser extensions, team collaboration, and provider marketplaces. A small curated provider set is fine; a provider marketplace is not.

## Premises
- Generic selection translation already exists. Winning on "can translate selected text" alone is not enough.
- The real daily pain is quality plus speed: users want output that is both correct and ready to send.
- Replacement actions matter as much as model quality. "Copy," "replace selection," and "re-optimize for email/chat/formal" are part of the product, not extras.
- Users will tolerate one-time accessibility permissions on macOS if the payoff is immediate.
- Provider abstraction is premature for v1. One strong translation path and one strong optimization path is enough.

## Approaches Considered
### Approach A: Native Selection Translator
Build a macOS menu bar app with a global hotkey, Accessibility-based text capture when possible, clipboard fallback when not, and a floating result panel near the cursor.

Pros:
- Directly serves the strongest use case: selected text in any app.
- Feels fast and native.
- Creates room for "replace selection" and workflow shortcuts.

Cons:
- macOS accessibility integration is fiddly.
- Some apps will resist direct text replacement, so fallbacks are required.

### Approach B: Clipboard Translator With Optimization
Build a simpler clipboard watcher that reacts whenever users copy text and shows a translation/optimization result.

Pros:
- Faster to implement.
- Less OS integration risk.

Cons:
- Feels noisier and less intentional.
- Adds friction because copy becomes part of every flow.
- Weaker wedge versus existing utilities.

## Recommended Approach
Choose Approach A, but implement it pragmatically:

- Primary path: global hotkey reads selected text and opens a compact translator panel.
- Fallback path: if direct selection read fails, SnapLingo tells the user to press copy once and instantly continues from the clipboard.
- Result panel is a copy-first surface with a primary `Copy` action, a visible mode switch, and a lightweight `Retry`.
- Optimization should not be a separate screen. It should be a one-click variation on the same result.

This keeps the product honest. The app is for fast in-flow desktop work, not for managing translation projects.

## Core User Flow
1. User selects text in any desktop app.
2. User presses the SnapLingo hotkey.
3. SnapLingo captures the selection directly or falls back to the clipboard path.
4. A compact panel appears with:
   - a segmented `Translate | Polish` mode switch
   - the current best result as the primary visual anchor
   - a muted, expandable original-text preview
   - explicit partial-state messaging when a better result is still loading
5. User finishes with one action: `Copy` or `Esc`.

The loop should feel closer to Spotlight than to a full editor.

## Non-Goals For v1
- Full document translation
- OCR from screenshots or images
- Voice input and speech output
- End-user provider marketplaces or arbitrary model switching
- Replace selection and undo
- Shared glossaries, team workspaces, or admin controls
- Mobile and Windows support before the macOS loop feels excellent

## Open Questions
- Which language pairs matter first? If the initial audience is Chinese-to-English knowledge work, the optimize feature can be much sharper.
- Is offline translation a real requirement, or is "fast cloud-first with local caching" acceptable for v1?

## Success Criteria
- A user can select text in a normal desktop app, press one hotkey, and get a result without manually opening another app.
- At least 80 percent of successful sessions finish with `Copy` in under 5 seconds end to end.
- Users choose `Optimize` often enough to prove it is core behavior, not decoration. A healthy target is 25 percent or more of translation sessions.
- Early users describe the app as "faster than opening DeepL/ChatGPT" and "good enough to send."

## Distribution Plan
- Deliverable: signed macOS app distributed directly from GitHub Releases or a simple product site.
- Packaging: notarized `.dmg` or `.zip`.
- Update path: Sparkle or a lightweight in-app update check after the core loop works.
- CI/CD: GitHub Actions builds, signs, and publishes tagged releases.

## Next Steps
1. Lock the first audience. Recommended starting point: Chinese-speaking developers, founders, and operators writing in English every day.
2. Define the exact v1 command set:
   - `Translate selection`
   - `Polish selection`
   - `Copy current result`
3. Build a clickable prototype of the floating panel and interaction timing before polishing the provider settings UX.
4. Validate five real daily workflows:
   - Slack message
   - email reply
   - browser article paragraph
   - document sentence rewrite
   - subtitle or short caption translation
5. Keep provider support curated and adapter-based: support a few strong providers, not an open-ended marketplace.

## Mini Design Spec
- Overall tone: `Spotlight / system utility`, not an AI chat widget
- Typography: `SF Pro` with explicit hierarchy, not default system sizing by accident
- Structure: single-column result card, segmented mode switch on top
- Result state: `Quick Translation` in partial state, `Polished Version` when optimization completes
- Original text: low-contrast, single-line preview with expand affordance
- Primary action: `Copy`
- Secondary actions: mode switch and `Retry`
- Success feedback: brief copied confirmation, then auto-dismiss
